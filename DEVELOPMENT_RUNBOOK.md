# DEVELOPMENT_RUNBOOK

## 0) 이 문서 목적
이 프로젝트에서 `npm run dev` / `lint` / 이미지(assets) 작업이 반복적으로 불안정해진 원인을 줄이기 위한 **운영 원칙 + 체크리스트**를 기록합니다.

---

## 1) 왜 dev 문제가 반복되었는지 (요약)
- **실행 엔진/캐시가 섞일 때**: Turbopack/Webpack 환경이 섞이거나 `.next` 캐시가 누적되면, 시작 단계(`Starting...`)에서 멈추거나 panic/cache 계열 에러가 반복될 수 있습니다.
- **의존성/내부 경로가 불완전할 때**: `next`, React 내부 파일, Tailwind 내부 경로 등을 “찾지 못하는” 오류는 대개 `node_modules`가 완전/일관 상태가 아니거나(부분 설치/버전 불일치) 캐시/번들 결과가 꼬인 경우에 함께 나타납니다.
- **검사(ESLint)와 실행이 겹칠 때**: 특정 파일 검사에서 멈추는 현상은, 실행/캐시가 이미 흔들리는 상태에서 병목이나 타임아웃처럼 증폭될 수 있습니다.

---

## 2) Webpack 고정 원칙
- **반드시** 개발 서버는 `package.json`의 `dev` 스크립트를 통해 실행합니다.
  - 사용 명령: `npm run dev`
- 직접 `next dev` 같은 우회 실행은 피합니다(모드/엔진이 달라질 수 있음).
- 포트가 이미 사용 중이면(`EADDRINUSE`) 먼저 기존 프로세스를 정리합니다.

---

## 3) 터미널 역할 분리 원칙
- **한 터미널 = 한 역할**로 유지합니다.
  - 터미널 A: `npm run dev`만 실행
  - 터미널 B: 필요 시 `npm run lint` / `npm run build`만 실행(가능하면 dev를 먼저 내린 뒤 실행)
- **동시에 여러 빌드/린트/데브를 돌리지 않습니다.**

---

## 4) 한 번에 한 섹션만 수정
- 화면/레이아웃/이미지 슬롯은 변화 범위가 작을수록 안정적입니다.
- 작업 단위:
  - “홈 섹션 1개”, “컴포넌트 1개”, “이미지 슬롯 1그룹”처럼 **딱 한 덩어리만** 변경합니다.
- 이후 반드시 화면 확인을 하고, 문제가 생기면 그 덩어리부터 되짚습니다.

---

## 5) slots 이미지 운영 원칙
- 이미지는 가능하면 **고정 슬롯 파일명**으로 운영합니다. (코드 수정 없이 파일만 교체)
- 실제 참조 경로는 `public/slots/*` 아래입니다.
  - 예: `/slots/home-hero-visual.png`, `/slots/home-workshop-banner.png`
- 비율/잘림 원칙:
  - `object-fit: cover` 계열: 가장자리가 일부 잘릴 수 있으니 **중앙(핵심 요소)** 위주로 배치
  - `object-fit: contain` 계열: 잘리지 않지만 여백이 생길 수 있으니 **프레임 가시성**(너무 빈 여백 방지) 확인
- 권장 비율(현재 코드의 wrapper 기준):
  - 아이콘(퀵메뉴): `1:1` (예: 1024x1024)
  - 자료실 썸네일: `4:3` (예: 1600x1200)
  - 홈 전자책 표지: `4:5` 계열(예: 1600x2000)
  - 워크샵 배너: `16:9` (예: 1600x900 또는 1920x1080)
  - 세로 후기 썸네일: `9:16` (예: 1080x1920)

---

## 6) 작업 시작 전 체크리스트
- `npm run dev`는 **이미 떠 있지 않은지**(포트/프로세스) 확인
- 변경은 “한 덩어리”만 수행(다른 섹션/파일 동시 변경 금지)
- dev 서버에서 작업할 경우:
  - 다른 터미널에서 동시에 `build/lint`를 돌리지 않기
- 이미지 교체는 `public/slots/`의 **고정 파일명**을 따르기

---

## 7) 서버 문제 발생 시 진단 순서
1. **포트 문제**: `EADDRINUSE`면 기존 프로세스가 점유 중인지 확인
2. **의존성/경로 문제**:
   - `next: command not found`
   - `Cannot find module ...react.development.js`
   - `Cannot find module ...tailwindcss/dist/lib.js`
   -> 이 경우는 “코드/레이아웃”보다 먼저 `node_modules` 상태와 버전/경로 일관성을 의심
3. **실행 엔진/캐시 문제**:
   - Turbopack panic, Starting... 장시간 멈춤, cache 관련 메시지
   -> `.next` 캐시 누적/혼재 가능성 점검
4. **검사 병목**:
   - eslint가 특정 파일 검사에서 멈추면, 실행 엔진/캐시 문제를 먼저 해결했는지 확인

---

## 8) 체크포인트 저장 규칙
- “안전한 한 단계”를 끝냈을 때만 코드 변경을 묶고 확인합니다.
- 각 체크포인트마다 아래를 기록합니다.
  - 무엇을 바꿨는지(섹션/컴포넌트)
  - 서버 동작 상태(`dev`가 Ready까지 갔는지)
  - 문제 발생 시 첫 로그(가장 먼저 나온 에러 1줄)
- 같은 이슈를 반복할 때는 “수치 미세조정”보다 **레이아웃/구조 변경 여부**와 **실행 환경 상태**를 우선 함께 기록합니다.
